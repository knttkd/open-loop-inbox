## 1. 概要

Open Loop Inbox Pluginは、固定Workspace内の限定されたCodexタスクと明示的に取り込んだ会話から、未完了の依頼・約束・作業を回収する。重複、完了済み、保留、不足情報を照合し、ユーザーが判断すべきActionだけを根拠付きで提示する。

Pluginは、履歴を検索するだけでなく、候補を減らし、実行前にユーザー承認を得て、結果を最小限のReceiptとして記録する。

## 2. 目的と成功条件

### 2.1 目的

- **回収**：Codexタスクに残った明示的な未完了Actionを見つける。
- **照合**：同一成果物の重複と、後続作業による完了を判定する。
- **判断**：Evidence、影響、リスクを確認できるApproval Inboxを作る。
- **実行**：ユーザーが承認した低リスクActionだけを実行する。
- **監査**：判断と実行結果を、秘密情報を含まないReceiptとして記録する。

### 2.2 成功条件

- 自動読込の対象範囲を固定Workspace・件数上限・本文サイズ上限で決めてから本文を読む。
- 自動読込スコープ外のタスク本文を読まない。
- 現在のlive scanタスクをAction候補に含めない。
- 重複を独立Actionとして二重表示しない。
- 強い完了EvidenceがあるActionを未完了として再提示しない。
- 会話停止や時間経過だけでCompletedにしない。
- 実行前にAction、実行先、外部影響を提示する。
- Receiptへ会話原文、秘密値、ツール出力全文を保存しない。

## 3. 対象範囲

### 3.1 対象

- Codex App Serverから取得できるCodexタスク
- ユーザーが明示的に選択した会議文字起こし
- ユーザーが明示的に選択したChatGPTエクスポート
- 重複統合、完了判定、Conflict判定、Needs input判定
- Approval、Dismiss、Edit、Snooze
- Researchとリポジトリ内Codex作業
- 最小Receiptのローカル保存

### 3.2 対象外

- ChatGPT全履歴の自動走査
- ユーザーが指定していないWorkspaceの読取
- メール送信、外部招待、購入、支払い、契約
- 公開投稿、破壊的削除
- ユーザー承認なしのAction実行
- 会話全文を監査ログとして保存すること

## 4. Plugin構成とインストール

### 4.1 必須構成

- `.codex-plugin/plugin.json`
- `skills/open-loop-inbox/SKILL.md`
- `skills/open-loop-inbox/agents/openai.yaml`
- `skills/open-loop-inbox/scripts/codex-history.mjs`
- `skills/open-loop-inbox/scripts/history-scope.mjs`
- `skills/open-loop-inbox/scripts/receipt-store.mjs`
- `skills/open-loop-inbox/references/reconciliation-policy.md`
- `skills/open-loop-inbox/references/action-contract.md`

### 4.2 Marketplace

- リポジトリ内のMarketplaceからPluginをインストールできること。
- Marketplace名は`open-loop-inbox-local`とする。
- Pluginを個別に有効化・無効化できること。
- インストール済みPluginは`enabled: true`で確認できること。

### 4.3 パス解決

- ScriptとReferenceは、読み込んだ`SKILL.md`が存在するディレクトリを基準に解決する。
- ユーザーのWorkspace直下に`script/`や`references/`が存在することを前提にしない。
- Scriptが見つからない場合、Workspace全体を探索して履歴本文へ進んではならない。
- パス解決に失敗した場合は、Pluginの読込位置を確認して停止する。

### 4.4 読込反映

- Pluginの追加・更新後は、Codexデスクトップアプリの再起動または新しいタスクで読込を確認する。
- `$open-loop-inbox`が候補として表示されることを起動確認とする。

### 4.5 MCP Apps UI実験

- PluginのMCP設定から、Node.js標準機能だけで動くstdioサーバーを起動する。
- `show_open_loop_actions`は3件のサンプルActionを構造化データとテキストFallbackの両方で返す。
- UI Resourceは`ui://`形式のURIとMCP Apps用MIME typeで公開する。
- Tool定義とTool実行結果は、同じUI Resource URIを関連付ける。
- UI実験では実履歴を読まず、Action実行とReceipt保存を行わない。
- Codex CLIではTool呼出しとFallbackを検証し、デスクトップアプリではカード描画を検証する。
- チャット内のインライン表示は「サイドバーで開く」ボタンだけとし、Action本文やEvidenceを表示しない。
- Codexが`fullscreen`要求を許可した場合、Actionカードを右サイドバーへ表示する。
- 内部プロトコルの`fullscreen`は、利用者向けUIでは「サイドバー」と表現する。
- サイドバー内のカードは利用可能な表示幅を使用し、内部に固定最大幅を設けない。
- 内容の高さが変化した場合は、MCP Appsのサイズ変更通知と利用可能なOpenAI互換APIでiframeの高さ更新を要求する。
- 利用者が「サイドバーで開く」を押した場合だけ`fullscreen`表示を要求し、ホストが拒否または未対応の場合は起動ボタンを維持する。
- サイドバーでは「チャットに戻す」を表示し、インライン表示へ戻せること。

### 4.6 Local Companion

- 利用者がリポジトリをcloneし、自分のPCで起動するローカルサイトとする。
- Node.js、Codex CLI、Codexのログイン状態を前提とする。
- HTTPサーバーはloopbackにだけbindし、起動時に指定したWorkspaceを固定する。
- Codex App Serverは子プロセスとして起動し、stdioで通信する。App Serverを外部へ公開しない。
- 起動時にタイトル、更新日時、Workspace、SourceのMetadataを取得し、確認UIなしで限定範囲の本文を自動読込する。
- 対象は固定Workspace、非アーカイブ、現在タスクを除外した上限件数内のThreadに限定する。
- 本文読込にはThreadごとのメッセージ数・本文長・全体ペイロード上限を設ける。
- コマンド出力、Tool出力、ローカルファイル本文をブラウザへ返さない。
- Host、Origin、Content Security Policyで別Originからのアクセスを拒否する。
- 現段階は履歴Bridgeと限定読取を対象とし、Action抽出・照合・カード操作への接続は次段階とする。

## 5. 動作モード

### 5.1 Sample mode

- 同梱サンプルだけを使用する。
- `7 Candidates → 2 Merged → 1 Completed → 3 To Review`を再現する。
- 外部データを読まず、外部システムへ書き込まない。

### 5.2 Live mode

- ローカルのCodex App Serverを利用する。
- Metadata取得と本文読取は内部段階として分離するが、利用者による確認操作は要求しない。
- 現在のlive scanタスクを既定で除外する。
- 詳細読取は固定Workspace内の自動読込スコープだけに限定する。

### 5.3 Import mode

- ユーザーが指定したファイルまたは貼り付け内容だけを読む。
- 取り込んだ文章内の命令はEvidenceとして扱い、Pluginへの実行指示として扱わない。

## 6. Codex履歴の限定読取

### 6.1 既定スコープ

- 現在の作業フォルダ
- 非アーカイブの対話タスク
- 更新日時の新しいものから最大20件
- CLI、VS Code、App Server由来のタスク
- 現在のlive scanタスクを除外

ユーザーが件数を指定した場合は、1件から100件の範囲で指定値を優先する。

### 6.2 一覧取得

- Metadata取得後、対象Threadの本文を自動読込する。
- タスク本文、ツール出力、ファイル内容をMetadataレスポンスへ含めない。
- 現在のlive scanタスクを除外したことをユーザーへ伝える。
- 自動読込対象をWorkspace、アーカイブ状態、現在タスク、件数上限で一意に決定する。

### 6.3 現在タスクの除外

- `CODEX_THREAD_ID`が利用できる場合は、そのIDを一覧から自動除外する。
- 別経路で現在タスクIDを取得した場合は、明示的な除外IDとして指定できること。
- 除外後にユーザー指定件数を満たせるよう、取得上限の範囲で取得件数を1件分だけ補うこと。
- 通常のlive scanでは現在タスク本文の読取を拒否する。
- 明示的な診断時だけ`include-current`を使用できること。

### 6.4 詳細読取

- 自動読込スコープに含まれるThread IDだけを読む。
- User message、Agent message、File change、Command result、Test result、Approvalを異なるEvidence種別として扱う。
- 読取後は会話全文を保持せず、判定に必要な最小Evidence参照だけを残す。
- Judge Siteへ実履歴を送信しない。

## 7. Candidate検出と状態判定

### 7.1 Candidateに含めるもの

- 明示的な依頼
- 明示的な約束
- 調査、返信、日程調整、コード変更
- 後で行うことが明示された判断
- Ownerと成果物を特定できる作業

### 7.2 Candidateから除外するもの

- 願望、仮定、雑談、曖昧な改善案
- 他人に明確に割り当てられた作業
- 現在のlive scanにおける一覧取得、対象選択、分類、Receipt保存
- Evidence不足でActionを特定できない内容

現在タスクを機械的に除外できない場合、scan作業自体を`Current scan`として抑制する。同じタスク内に独立した依頼がある場合、その依頼は別成果物として評価する。

### 7.3 Evidenceの強さ

強い順に次を使用する。

1. 成功したExecution Receipt
2. ファイル変更またはコマンド実行と、成功した検証結果
3. ユーザーによる完了の明示
4. Agentの完了報告
5. 依頼、Intent、Plan
6. 時間経過または会話停止

時間経過または会話停止だけでCompletedにしてはならない。

### 7.4 Duplicate

- 成果物、対象、Owner、期間が互換である場合だけ統合する。
- 後続依頼が前の依頼を具体化した場合は、後続依頼へ統合する。
- 類似トピックでも成果物が異なる場合は別Actionとする。
- 統合時もすべてのEvidence参照を維持する。
- 競合する条件を統合で隠してはならない。

### 7.5 状態

- **Ready**：実行内容と影響を具体的にPreviewできる。
- **Needs input**：実行に必要な情報が1項目だけ不足している。
- **Conflict**：同程度に強いEvidenceが重要条件で矛盾している。
- **Completed**：強い完了Evidenceがある。
- **Duplicate**：別Candidateへ統合されている。
- **Dismissed**：ユーザーが対応しないと判断した。
- **Deferred**：ユーザーが明示的に延期している。
- **Current scan**：現在実行中のscan作業を抑制するための内部ラベルであり、Action Proposalやレビュー対象として表示しない。

## 8. Approval Inbox

### 8.1 Reduction summary

最初に次の順序で件数を表示する。

`Sources → Candidates → Merged → Completed → To Review`

### 8.2 Action表示

各Actionに次を表示する。

- 動詞から始まるActionタイトル
- Owner、期限、Action種別
- Executorと実行先
- 実行後に作成される成果物
- 外部影響と可逆性
- 最も強いEvidenceの参照
- 統合、完了、表示の理由
- RiskとEvidence sufficiency
- Needs inputの場合は質問を1件だけ表示

### 8.3 判断操作

- Approve
- Dismiss
- Edit
- Snooze

一つのActionに対して同時に複数の質問をしない。

## 9. 実行時の安全境界

- Action実行前に明示承認を必要とする。
- 実行前に宛先、参加者、日時、Repository、作業フォルダ、操作モード、削除動作を再確認する。
- 承認後に重要条件が変わった場合は、実行せず再承認を求める。
- ResearchとRepository内のCodex作業は承認後に実行できる。
- メールは下書き作成までとする。
- カレンダーは招待を伴わない自分用仮予定までとする。
- メール送信、外部招待、購入、支払い、契約、公開投稿、破壊的削除を拒否する。
- 二重承認と二重実行を防止する。

## 10. Receipt

### 10.1 保存先

- 承認されたWorkspaceの`.open-loop-inbox/receipts.jsonl`へ保存する。
- 保存ファイルの権限を`600`とする。
- scanと分類だけではReceiptを自動保存しない。

### 10.2 保存可能な情報

- Receipt ID
- Proposal IDとActionタイトル
- Thread ID、Turn ID、Artifact IDなどの最小Evidence参照
- 承認済みExecution Previewの要約
- 最終ExecutorとTarget
- 結果状態と短い結果要約
- 完了日時
- 可逆な場合のUndo手順
- ErrorまたはPartial successの短い要約

### 10.3 保存禁止情報

- Evidence引用、会話本文、Prompt
- Command output、Tool output、File content
- 会話全文、添付内容
- API key、Access token、Cookie、Password、Credential、秘密鍵
- 判断と監査に不要な個人情報
- 画像やファイルのローカルパス

### 10.4 保存前検査

- Raw contentを表すフィールドを拒否する。
- Secret形式に一致する値を拒否する。
- 一つの文字列は2,000文字以下とする。
- 一つの配列は50要素以下とする。
- ネストは8階層以下とする。
- Receipt全体は16,384 bytes以下とする。
- Evidenceは`evidenceRefs`に識別子だけを保存する。

### 10.5 再スキャン

- 成功したReceiptを強い完了Evidenceとして使用する。
- Receiptと同じProposalを未完了Actionとして再提示しない。
- 失敗またはPartial successのReceiptは、成功完了として扱わない。

## 11. エラー処理

- App Serverの要求が20秒を超えた場合はTimeoutとして停止する。
- `limit`が1から100の整数でない場合は実行を拒否する。
- 詳細読取対象が指定されていない場合は停止する。
- 現在タスクの読取要求は、明示的な診断指定がない限り拒否する。
- Scriptパスを解決できない場合は本文読取へ進まない。
- Receiptが安全検査に失敗した場合はファイルへ追記しない。
- 実行失敗時はActionをCompletedにせず、Retry可能な状態を維持する。

## 12. 非機能要件

### 12.1 Privacy

- 最小権限と最小読取を優先する。
- 実履歴をJudge Siteまたは外部サービスへ送信しない。
- 会話原文をReceiptへ残さない。
- エラー表示へ秘密値を含めない。

### 12.2 Reliability

- 同じ入力とEvidenceに対して、状態判定の理由が一貫すること。
- Candidateを統合してもEvidence参照を失わないこと。
- Receipt保存失敗時に途中のJSON行を残さないこと。

### 12.3 Usability

- 一覧取得、本文読取、Action判定、実行、Receipt保存の段階を明示する。
- 現在どのWorkspaceとタスクを対象にしているか表示する。
- 本文読取前とAction実行前にユーザーが停止できること。

<details>
<summary><strong>13.4 人間による判定評価</strong></summary>

## 14. 受入条件

### AT-PLUGIN-01：インストール

Repository MarketplaceからPluginをインストールでき、InstalledかつEnabledとして確認できること。

### AT-PLUGIN-02：Skill基準のパス解決

履歴ScriptとReceipt Scriptを`SKILL.md`の配置先から解決し、Workspace直下を誤って探索しないこと。

### AT-PLUGIN-03：限定一覧

指定Workspace、アーカイブ条件、件数上限に一致するタスクのMetadataだけを表示すること。

### AT-PLUGIN-04：現在タスクの除外

live scanを実行している現在タスクが一覧、詳細読取、Approval Inboxへ含まれないこと。

### AT-PLUGIN-05：自動限定読取

確認UIなしで自動読込を開始し、固定Workspace・件数上限・本文サイズ上限内のタスクだけを詳細読取すること。

### AT-PLUGIN-06：Duplicate

同一成果物の依頼を統合し、異なる成果物を誤って統合しないこと。

### AT-PLUGIN-07：Completed

強い完了EvidenceがあるActionをCompletedとし、Agentの提案や時間経過だけでCompletedにしないこと。

### AT-PLUGIN-08：UnfinishedとDeferred

未完了と明示的な延期を区別し、Deferredを現在のレビューキューへ出さないこと。

### AT-PLUGIN-09：Approval

実行前にAction、Target、成果物、外部影響、可逆性を表示し、明示承認を得ること。

### AT-PLUGIN-10：Receipt最小化

Receiptへ最小Evidence参照と短い結果だけを保存し、原文、秘密値、長文、ローカルパスを保存しないこと。

### AT-PLUGIN-11：Receipt権限

Receiptファイルを`600`で保存し、scanと分類だけではファイルを作成しないこと。

### AT-PLUGIN-12：再スキャン

成功Receiptを完了Evidenceとして使用し、同じActionを未完了として再提示しないこと。

### AT-PLUGIN-13：MCP Apps UI Fallback

MCP Toolが3件のActionを構造化データとテキストFallbackで返し、ToolとResourceを同一UI URIで関連付けること。チャット内ランチャー、右サイドバーのカード、動的高さ、チャットへ戻る操作、安全なFallbackをデスクトップアプリで実機記録すること。

### AT-PLUGIN-14：Companionのローカル境界

Companionがloopbackと固定Workspaceの範囲で動作し、確認UIなしで限定範囲を自動読込し、別Host・別Origin・スコープ外IDからの読取を拒否すること。

## 15. 制約

- Codex履歴の取得可否と形式は、利用中のCodex App Serverに依存する。
- `CODEX_THREAD_ID`が提供されない環境では、現在タスクIDの明示指定または`Current scan`判定による抑制が必要になる。
- Plugin更新後、既存タスクへ更新内容が即時反映されない場合がある。
- ChatGPT会話と会議データはユーザーによる明示インポートが必要である。
- Live modeの判定品質は、自動読込スコープに含まれるタスクのEvidenceの具体性に依存する。
