# Open Loop Inbox（未完了事項インボックス）実装フェーズ

## 文書目的

この文書は、Open Loop Inbox（未完了事項インボックス）の実装順と各フェーズの完了条件を定める。プロダクト要件は「何を満たすか」、本書は「何をどの順番で作るか」を扱う。

## Phase 0：実装方針とスコープの固定

### 成果物

- ログイン不要で試せるCodex Siteの公開デモ
- 実際のCodexチャット履歴を選択して読めるCodex Plugin

### 確定した境界

| 項目 | Codex Site | Codex Plugin |
|---|---|---|
| 入力データ | 架空の同梱サンプルのみ | ユーザーが明示選択した実Codexチャット履歴。必要に応じて明示インポートしたログ |
| 実ユーザー履歴 | 扱わない | 対象スレッドと期間を表示してから読み取る |
| Research | Demo Mock | 実行し、Plugin専用ローカル保存先へMarkdownレポートを作成 |
| Gmail Draft | Demo Mock | Demo Mock。Gmailアカウントは接続しない |
| 会議ドラフト | Demo Mock | Demo Mock。Calendarアカウントは接続せず、自分用の仮予定だけを表現する |
| 共通部分 | 対応、根拠、対応記録、照合・整理のルール、実行キューの契約 | 同左 |

### 初期版の実行操作

すべてのActionで、内容に依存しない次の2つの実行操作を提供する。

1. `そのまま実行`：カードの現在の実行Previewを使う。
2. `指示を追加して実行`：ユーザーの追加指示をPreviewへ反映してから実行する。

どちらを選んでも直ちには実行せず、30秒の猶予を持つ実行キューへ入れる。猶予中のUndoはキューから取り消し、実行するかどうかを判断する状態へ戻す。実行後の複雑なロールバックは初期版では行わない。

`実行しない`と判断したActionは、時間制限なくUndoでき、実行可否を判断する状態へ戻せる。

### Phase 0の完了条件

- SiteとPluginの入力境界、実行境界、保存先が本書とプロダクト要件で一致している。
- Research、Gmail Draft、会議ドラフトの3種類のサンプル実行が定義されている。
- 実行キュー、30秒の猶予、Undoの状態遷移が定義されている。

## Phase 1：Shared Coreとサンプルシナリオ

### 実施内容

- `Source`、`EvidenceSpan`、`CandidateAction`、`ActionProposal`、`ApprovalDecision`、`ExecutionReceipt`の共通スキーマを定義する。
- 実行キュー用に、実行方法、追加指示、キュー投入時刻、実行予定時刻、キュー状態を保持する。
- 架空のCodexスレッド3件、文字起こし1件、外部会話ログ1件を同梱する。
- 重複統合、部分完了、追加確認、実行可能な下書きの期待結果をJSONで定義する。

### 完了条件

- サイトとプラグインが同じ対応・根拠・対応記録のスキーマを利用する。
- 作業実行の状態と対応事項の状態を別々に判定できる。
- ゴールデンデータで重複統合と部分完了を自動テストできる。

## Phase 2：Codex Siteの公開デモ

### 実施内容

- `Try a sample workday`から始まるサンプル専用フローを実装する。
- 対応カード、根拠の詳細、承認／対応不要／編集、スワイプ、キーボード操作、取り消しを実装する。
- Demo Research、Demo Gmail Draft、Demo Meeting Draftをモック実行し、Receiptを表示する。
- `そのまま実行`と`指示を追加して実行`の双方を、30秒猶予つき実行キューへ接続する。

### 完了条件

- 外部認証・実アカウント・実ユーザー履歴なしに、Receiptまで到達できる。
- 30秒以内のUndoで、実行されずカードが判断状態へ戻る。
- `実行しない`判断は、いつでもUndoできる。

## Phase 3：Codex Pluginの実履歴・Research実行

### 実施内容

- 対象期間とスレッドをユーザーが選べる履歴取得を実装する。
- 選択済みCodex履歴を共通Artifactへ正規化し、Shared Coreで照合する。
- 承認されたResearchを実行し、Plugin専用ローカル保存先へMarkdownレポートを作成する。
- Researchの保存先、実行内容、結果、残条件をReceiptへ残す。
- Gmail Draftと会議ドラフトはPlugin内でもDemo Mockに留め、外部アカウントを接続しない。

### 完了条件

- ユーザーが選択した実Codexスレッドから少なくとも1件のActionを生成できる。
- Researchレポートが指定のローカル保存先へ作成され、Receiptから確認できる。
- Pluginは選択外のCodex履歴を読まない。

## Phase 4：品質保証と公開

### 実施内容

- 公開Siteのログイン不要・モバイル・キーボード・Reduced Motion対応を確認する。
- 実行キューの二重実行、30秒Undo、Dismissの長期Undoをテストする。
- Evidenceの追跡、重複統合、部分完了、Closed除外をゴールデンデータで検証する。
- PluginのResearch保存とReceiptを実データで検証する。

### 完了条件

- 公開Siteでサンプルフローを2分以内に完了できる。
- Gmail／Calendarへの実アカウント接続・送信・招待が発生しない。
- Plugin Researchの成果物とReceiptを確認できる。
