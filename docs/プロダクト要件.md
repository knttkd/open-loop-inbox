# Open Loop Inbox
**Codexの複数スレッドと、取り込んだ会話ログに散らばった「やり残し」を、一つの実行キューへ。**

Open Loop Inbox（未完了事項インボックス）は、個別作業の実行が終わったかだけではなく、その作業を生んだ対応事項が本当に満たされたかを追跡する。

重複、条件変更、部分完了、承認待ちを横断して照合し、今も人間の判断が必要なActionだけを提示する。

## 1. Why：解く課題とプロダクト仮説

### 1.1 課題定義：未完了事項の分散

Codexで進める仕事は、一つのスレッドだけに収まらない。
あるスレッドで調査を始め、別のスレッドで条件を変え、さらに別のスレッドで実装とテストを行う。
会議の文字起こしや外部チャットをCodexへ取り込めば、返信、共有、承認といった約束も同じ作業空間へ入る。

問題は、履歴が多いことだけではない。
同じ依頼が別の表現で重複し、後のスレッドで一部だけ完了し、さらに条件が更新される。
ユーザーは、それらを自分で読み比べなければならない。

ここでは二つの完了を区別する。
**個別作業の実行**は、テスト成功、ファイル変更、レポート生成など、エージェントへ渡した個別作業の完了を示す。
**対応完了までの流れ**は、その作業を生んだ元の対応事項が、共有、返信、人間の承認、外部への反映まで含めて満たされたかを示す。
個別作業の実行が終わっても、対応完了までの流れは未完了のまま残り得る。

> **解くべき中心課題**
>
> Codex内の複数スレッドと取り込んだ会話ログを読み返さず、「どの対応事項がまだ未完了で、今どの対応を判断すべきか」を短時間で把握できないこと。

### 1.2 現状のユーザー行動

1. Codexのスレッドや外部会話の中で、次の作業や約束が生まれる。
2. ユーザーは別の作業へ移るため、その場でToDoへ登録しない。
3. 後で複数のCodexスレッドと取り込んだログを開き、何を依頼し、どこまで終わったかを確認する。
4. 同じ依頼を重複して登録するか、個別作業の実行が終わっただけの作業を対応事項の完了と誤認する。
5. 返信、共有、承認などの残作業が見えず、重要なフォローアップが抜ける。

### 1.3 問題を構成する5つの摩擦

| **摩擦**   | **現象**                                                                 | **結果**                                      |
|------------|--------------------------------------------------------------------------|-----------------------------------------------|
| 発見コスト | 複数のCodexスレッドと取り込んだ会話ログを一つずつ開く。                 | 未完了事項を見つける前に確認を打ち切る。      |
| 照合コスト | 同じActionが別スレッドや外部ログに異なる表現で残る。                    | 重複タスクと古い条件が残る。                  |
| 状態コスト | テストやファイル変更は完了しているが、共有や承認は終わっていない。      | 作業実行の完了を対応事項の完了と取り違える。   |
| 判断コスト | 抽出候補が多く、チャットで一件ずつ根拠と影響を確認する。                | レビューキュー自体が新しい仕事になる。        |
| 実行コスト | 残作業を把握した後、別のツールへ内容を再入力する。                      | 「分かったが進めていない」状態が残る。        |

### 1.4 原因仮説

- Codexの各スレッドは個別の作業には強いが、別スレッドの対応事項を自動的に一つの状態へ統合することを主目的にしていない。
- 外部の文字起こしやチャットログは、Codexへ取り込んでも、後続の実行結果と自動的には結び付かない。
- 一般的なタスク管理は、ユーザーがタスクを登録した後から始まるため、会話に残った約束の回収を人間に委ねる。
- Loop Engineeringは個別作業の実行を進められるが、複数の人、会話、エージェントをまたぐ対応事項の重複、変更、承認待ちを外側で整理する層が必要になる。
- ユーザーは、毎回プロンプトで全履歴の照合を依頼するより、定期的に開けるInboxで少数の判断だけを処理したい。

### 1.5 プロダクト仮説

> **主仮説**
>
> Codex内の複数スレッドと、ユーザーが明示的に取り込んだ会話ログを横断し、対応候補を回収して対応事項単位で照合・整理すれば、ユーザーは「今判断すべき少数」だけを安全に処理できる。
> 承認後の実行結果を対応記録として元の対応事項へ戻せば、作業実行の完了と対応事項の完了を区別しながら、同じ未完了事項の再提示を減らせる。

| **仮説ID** | **検証する仮説**                                                                                   | **反証条件**                                                              |
|------------|----------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------|
| H1         | 横断照合により、単純抽出よりレビュー件数を30%以上削減できる。                                      | 重複統合、部分完了、Closed除外がほぼ発生せず、単一スレッド抽出と差がない。 |
| H2         | 根拠、残条件、実行先、影響を一枚に表示すれば、チャット形式より判断時間が短い。                      | 詳細を毎回開かないと判断できず、チャットと同等以上に時間がかかる。        |
| H3         | 右スワイプを実行承認に結び付けると、把握から実行までの離脱が減る。                                 | 誤実行への不安が強く、ユーザーが承認を避ける。                            |
| H4         | Codex履歴を情報源と実行先の両方にすれば、開発作業の実行結果を元の対応事項へ自然に結び戻せる。     | 履歴取得、部分完了判定、対応記録の照合精度が低く、Codex利用が装飾に見える。 |

### 1.6 解かない課題

- 常時録音デバイスそのものの開発。
- 録音、文字起こし、話者分離のエンジン開発。
- すべての発言を完全に記憶して検索するライフログ。
- 万能な個人CRM、ナレッジベース、プロジェクト管理システム。
- ユーザー承認なしのメール送信、購入、公開投稿。
- ChatGPTを含む外部サービスの全履歴へ無条件にアクセスする仕組み。

## 2. Value：提供価値と成功指標

### 2.1 Value Proposition

> **一文の価値提案**
>
> Codex内の複数スレッドと取り込んだ会話ログに散らばった対応事項を照合・整理し、今判断すべき対応だけを、根拠と残条件を備えた実行キューとして提示する。

### 2.2 5段階の価値チェーン

| **段階**          | **ユーザー価値**                                               | **システム責務**                                                               |
|-------------------|----------------------------------------------------------------|--------------------------------------------------------------------------------|
| 回収    | どのスレッドで話したかを覚えていなくても、候補が集まる。       | Codex履歴と明示的に取り込んだ会話ログから未完了事項の候補を抽出する。          |
| 照合・整理  | 重複、条件変更、部分完了を自分で整理しなくてよい。              | 根拠を時系列で統合し、作業実行と対応事項の状態を分けて更新する。       |
| 確認・判断     | 少数のカードを見て、承認、編集、保留、対応不要を素早く決められる。 | 元の対応事項、完了済みの範囲、残条件、実行前の確認内容、リスクを一枚に表示する。       |
| 実行    | 別のツールへ再入力せず、承認後に仕事が進む。                    | プラグインによる調査、デモのGmail下書き、デモの会議下書きを実行する。             |
| 対応記録・完了   | 何が終わり、何が残ったかを確認できる。                          | 実行結果を元の対応事項へ戻し、完了済み、一部完了、要再対応を判定する。 |

### 2.3 Jobs to Be Done

- 一日の終わりに、複数のCodexスレッドを読み返さず、まだ未完了の対応事項だけを確認したい。
- 週次計画の前に、複数プロジェクトのCodex履歴と取り込んだ会話ログを短時間で照合したい。
- あるスレッドで実装が終わっている場合、元の依頼を再表示せず、共有や承認など残っている部分だけを見たい。
- Agentが提案した操作を、元の発言、残条件、影響を見てから安全に承認したい。
- タスク名だけでなく、調査条件、修正対象、返信文、予定候補まで実行準備された状態で受け取りたい。

### 2.4 Before / After

| **現状**                                         | **Open Loop Inbox（未完了事項インボックス）導入後**                                      |
|--------------------------------------------------|----------------------------------------------------------------|
| 複数のCodexスレッドと外部ログを開き、候補をメモへ転記する。 | 対象期間とインポート対象を選ぶと、対応事項の候補が一つのインボックスへ集約される。 |
| 同じActionが複数の表現で重複する。               | 複数のEvidenceを一つのActionへ統合し、「統合理由」を表示する。 |
| 作業実行が終わったか、対応事項まで完了したかを別々に確認する。 | 後続の根拠から一部完了と完了を分け、残っている対応だけを表示する。 |
| 残作業を見つけた後、別のツールへ再入力する。                     | 承認するとCodexや接続アプリが調査、修正、下書き、仮予定を作る。         |
| 完了判定と実行結果の関係が分からない。                           | 元の対応事項、根拠、完了済みの範囲、残条件、対応記録を表示する。   |

### 2.5 North Star Metricと補助指標

> **North Star Metric**
>
> 1分あたりに解決した未完了事項数：1分の確認・判断で、承認・対応不要・編集・完了確認まで解決できた未完了事項数。

| **指標**                 | **定義**                                                       | **初期版目標**         |
|--------------------------|----------------------------------------------------------------|---------------------|
| 確認対象の削減率    | 検出候補から重複・完了を除外し、実際の確認対象を減らした割合。 | 30%以上             |
| 対応候補の適合率         | 表示された対応のうち、ユーザーが妥当と判断した割合。         | 85%以上             |
| 根拠の付与率        | すべての対応に確認可能な根拠が付く割合。                     | 100%                |
| Median Decision Time     | カード表示から判断までの中央値。                               | 10秒以内            |
| Execution Completion     | 承認された低・中リスクActionがReceiptまで到達する割合。        | 95%以上（公開デモ環境） |
| Critical False Execution | 承認内容と異なる宛先・操作・送信が起きた件数。                 | 0件                 |

## 3. なぜCodexなのか

### 3.1 Codexは「開発に使ったツール」ではなく製品構成要素

| **役割**          | **本プロダクトでの責務**                                              |
|-------------------|-----------------------------------------------------------------------|
| Source            | 保存済みCodexスレッドから、依頼・未完了・完了・変更の証拠を取得する。 |
| 照合・整理機能        | Codex内の複数スレッドと取り込んだ会話ログを対応事項単位へ統合し、更新、部分完了、矛盾を判定する。 |
| 計画機能           | 曖昧な発言を実行可能な対応案へ変換し、不足情報を特定する。   |
| Executor          | 調査、コード修正、テスト、ファイル更新などを実行する。                |
| Permission Broker | 外部書込みや環境変更の前に承認を要求し、判断を実行境界に結びつける。  |
| 記録機能           | 元の対応事項、根拠、人間の判断、実行結果、残条件を対応記録として残す。                          |

### 3.2 公式機能との対応

Codex App Serverは、認証、会話履歴、承認、ストリーミングされたエージェントイベントを独自製品へ組み込むための公式インターフェースである。保存スレッドを一覧するthread/listと、ターン履歴を含めて読み取るthread/readが提供されるため、Codex履歴を実ソースとして扱える。[R1]

App Serverはコマンド実行やファイル変更の承認要求をクライアントへ送信できる。Open Loop Inboxの承認カードは、このPermissionモデルをユーザーが高速に扱えるレビューUIへ翻訳する。[R2]

CodexのSkillは反復可能な判断手順を保持でき、Pluginは複数Skillに加え、App mapping、MCP server設定、表示アセットを一つのパッケージにまとめられる。これにより、Open Loop検出ポリシー、照合規則、実行ツール依存を配布可能な製品単位にできる。[R3]

### 3.3 なぜ汎用LLM＋独自APIだけでは弱いのか

| **汎用構成の弱点**                                | **Codex採用で得る利点**                                                |
|---------------------------------------------------|------------------------------------------------------------------------|
| Codex履歴を別途エクスポート・同期する必要がある。 | App Serverの保存スレッド一覧・読取を使い、履歴を一次情報として扱える。 |
| コード実行・テスト・修正の実行基盤を自作する。    | Codexの既存実行環境、レビュー、承認、履歴を利用できる。                |
| 開発ワークフローの「完了」を外部から推測する。    | Codexターン内の実行・テスト・結果を完了証拠として照合できる。          |
| 承認UIと実行権限を別々に設計する。                | Codexの承認プロトコルとActionカードを結びつけられる。                  |
| 製品が「APIをつないだだけ」に見えやすい。       | Codexの履歴→判断→実行→Receiptという閉ループを示せる。                  |

### 3.4 Codexでなければ成立しない部分／Codexでなくても成立する部分

| **分類**            | **内容**                                                                                               |
|---------------------|--------------------------------------------------------------------------------------------------------|
| Codexとの強い必然性 | Codex履歴のライブ取得、作業実行の結果と対応事項の状態の判定、コードと調査の継続実行、承認と対応記録。 |
| 他基盤でも再現可能  | 明示インポートされた会話からのAction抽出、カードUI、Gmail下書き、カレンダー予定、意味的重複判定。       |
| 正しい主張          | Codexの履歴・Skill・Plugin・App Serverを使うことで、開発作業に特化した閉ループを製品として構成できる。 |
| 避ける主張          | Codexでなければ技術的に絶対実現できない。                                                              |

### 3.5 技術的境界

- Codex履歴：初期版でライブ取得する一次ソース。
- 外部会話ログ：会議文字起こし、音声メモの文字列、ChatGPTエクスポートなどを、ユーザーが内容を確認して明示インポートする。
- 外部履歴アクセス：ChatGPTを含む外部サービスの全履歴を自動走査するAPIは前提にしない。
- 録音処理：初期版は文字起こし済みデータを入力し、録音と話者分離は扱わない。
- スワイプUI：公開デモサイトで必須。Codex内では同じ対応案を承認、編集、対応不要にできれば初期版の中核要件を満たしたものとする。

## 4. 競合・代替手段と未充足領域

本章の競合評価は、2026年7月17日時点の各社公式公開情報に基づく。公開ページに記載がないことは、製品内に機能が存在しないことの証明ではない。以下は「公表された主目的・ワークフロー」と本プロダクト仮説の差分である。

### 4.1 競合カテゴリ

| **カテゴリ**            | **代表例**                       | **強み**                                                       | **本プロダクトとの主な差分**                                                          |
|-------------------------|----------------------------------|----------------------------------------------------------------|---------------------------------------------------------------------------------------|
| Ambient AI / wearable   | Bee、Omi                         | 日常会話を継続的に取り込み、要約、タスク、リマインダーへ変換。 | Codexスレッドの実行Evidenceと、明示インポートされた会話のCommitmentを照合することを主目的にしない。 |
| AI meeting intelligence | Otter、Granola                   | 会議の記録、横断検索、Action item、フォローアップ、外部連携。  | 会議後にCodexで進んだExecutionを、元のCommitmentの部分完了として照合することを主目的にしない。       |
| ChatGPTの履歴・Memory   | ChatGPT Search、Memory、Projects | 過去会話の検索・参照・プロジェクト文脈。                       | 監査可能な未完了台帳、重複統合、完了抑制、承認付き実行キューではない。                |
| Task manager            | Todoist、Asana等                 | 確定したタスクの管理、期限、担当、進捗。                       | タスクになる前の会話からOpen Loopを回収・照合する部分がユーザーまたは別自動化に残る。 |

### 4.2 主要競合の公式公開機能

| **製品** | **公表されている強み**                                                                        | **Open Loop Inboxが狙う未充足領域**                                                       |
|----------|-----------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------|
| Omi      | 会話からcommitmentやToDoをTaskとして保存し、Google Tasks、Todoist、Asana等へ連携できる。      | 取り込んだ会話のCommitmentとCodexスレッドの実行Evidenceを一つの状態へ照合すること。       |
| Bee      | 会話、タスク、場所などを要約・個人インサイト・適時リマインダーへ変換するAmbient personal AI。 | AI開発履歴を含むクロスソースのOpen Loop台帳、根拠付きの高速承認・実行。                   |
| Otter    | 会議を文字起こしし、会議横断検索、要約、Action item、連携・ワークフローを提供。               | 会議の後にCodexで進んだ作業をEvidenceとして取り込み、Commitmentの残条件を更新すること。   |
| Granola  | 会議ノート、Action、Memory、会議横断Chat、フォローアップ、Zapier連携。                        | 取り込んだ会議ログとCodexの複数スレッドを横断し、部分完了を残作業へ変換すること。          |
| ChatGPT  | キーワードによる履歴検索、過去チャット参照、Project内コンテキスト。                           | Open、Partially closed、Closed、DuplicateをEvidence付きで照合し、実行承認へつなぐ台帳ではない。 |

ChatGPTの履歴検索とMemoryは過去会話の検索・参照を支援するが、Open Loopの状態台帳や承認付き実行キューを主目的とはしていない。[R5, R6]

Omiは会話からTasksを作り、外部タスクマネージャへ連携することを公式に説明している。Otterは会議横断のAI Chat、Action item、接続アプリへの自動化を提供する。Granolaは会議横断の質問、Action、フォローアップ、Zapier連携を提供する。Beeは日常の会話・タスク等を要約、インサイト、リマインダーへ変換する。[R7, R8, R9, R10, R11]

### 4.3 「競合ではだめ」ではなく「競合が最適なケース」

| **ユーザーの目的**                                                             | **最適な選択**                                                        |
|--------------------------------------------------------------------------------|-----------------------------------------------------------------------|
| 会議を高品質に記録・要約したい                                                 | OtterまたはGranola。Open Loop Inboxは録音品質や議事録生成で競わない。 |
| 日常会話をウェアラブルで記録したい                                             | BeeまたはOmi。Open Loop Inboxはハードウェアを作らない。               |
| 確定済みタスクの進捗を管理したい                                               | 既存Task manager。Open Loop InboxはSystem of Recordを置き換えない。   |
| 過去の特定チャットをキーワードで探したい                                       | ChatGPT Search。Open Loop Inboxは検索UIではない。                     |
| Codexの複数スレッドと取り込んだ会話ログを横断し、重複、部分完了、条件変更を照合したい | Open Loop Inboxの主戦場。 |

### 4.4 差別化要件

| **差別化軸**           | **必須状態**                                                                      |
|------------------------|-----------------------------------------------------------------------------------|
| Cross-source           | Codex live、文字起こし import、外部会話 importの3種類を同じActionモデルへ正規化。 |
| 照合・整理         | 重複統合、部分完了、完了済みによる除外、期限と条件の更新、矛盾検出を実際に行う。          |
| 根拠を優先         | すべての対応に複数ソースの根拠を表示。推測と原文を区別。                        |
| 確認・判断の効率      | チャット往復ではなく、スワイプ・ボタン・キーボードで判断。                        |
| Intent-bound execution | ユーザーが承認した内容と実際の実行内容を一致確認し、差分は再承認。                |
| Codex-native loop      | Codex履歴を読み、Codexが完了証拠を見つけ、必要な作業をCodexが実行。               |

### 4.5 模倣されやすい部分／守るべき部分

| **模倣されやすい**                                      | **守るべきプロダクト資産**                                                                                            |
|---------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------|
| スワイプカード、ToDo抽出、一般的なGmail／Calendar連携。 | クロスソースAction Graph、完了・更新の照合ルール、Evidence品質、ユーザーの承認・却下学習、Codex作業状態との閉ループ。 |

## 6. プロダクト範囲と構成

### 6.1 製品構成

| **サーフェス**    | **目的**                               | **データ**                                            | **実行**                                           |
|-------------------|----------------------------------------|-------------------------------------------------------|----------------------------------------------------|
| Codex Plugin Mode | 本体。選択した実Codex履歴の取得、照合、承認、実行。 | ユーザーが明示選択したCodexスレッド。必要に応じて明示インポートされた会話ログ。 | ResearchはPlugin専用ローカル保存先へ実行。Gmail Draftと会議ドラフトはDemo Mock。 |
| Public Demo Mode   | 認証や外部データの準備なしでコア体験を試す。 | 架空の同梱サンプルのみ。実ユーザー履歴は扱わない。 | Demo Research、Demo Gmail Draft、Demo Meeting Draftによるモック。 |
| Shared Core       | 両面の判断一貫性。                     | 共通Source、Evidence、Action、Receiptモデル。         | 共通Risk policyとReconciliation policy。           |

### 6.2 初期版のゴール

- 複数ソースからOpen Loop候補を検出できる。
- 意味的重複を一つへ統合できる。
- 後続のCodex履歴に完了証拠があればレビュー対象から除外できる。
- 不足情報を一問だけ聞くActionとして提示できる。
- スワイプ／ボタンで承認・却下・編集できる。
- Pluginで承認したResearchを実行し、Plugin専用ローカル保存先へMarkdownレポートを作成できる。
- Gmail Draftと会議ドラフトは、外部アカウントを接続せずDemo Mockとして試せる。
- すべての判断と実行にEvidenceとReceiptが残る。

### 6.3 初期版の対象外

| **対象外**               | **理由**                                               |
|--------------------------|--------------------------------------------------------|
| 録音・文字起こしエンジン | 価値の中心ではなく、時間・品質リスクが高い。           |
| 外部サービス全履歴の自動取得 | 初期版は明示インポートで正確な権限境界を保つ。           |
| 商品購入・決済           | 誤実行の影響が大きく、中心価値に比べてリスクが高い。   |
| メール自動送信           | 下書きで価値を示せる。送信は二段階承認が必要。         |
| チーム共有・権限管理     | 初期版は個人利用を優先する。                                    |
| 高度な長期学習           | 承認・却下の履歴収集までは行うが、個人最適化は次段階。 |

### 6.4 優先順位

| **分類**      | **要件**                                                                                                       |
|---------------|----------------------------------------------------------------------------------------------------------------|
| Must          | Codex実履歴、明示インポートした文字起こし、明示インポートした外部会話ログの3 Source kind、候補検出、重複統合、部分完了、Closed除外、Evidence、スワイプ、モック実行、Receipt、公開URL。 |
| Should        | 不足情報の一問確認、実行キュー、Undo、Action confidence。                                                       |
| Could         | 期間フィルター、優先順位、User preference学習、複数言語、自動定期スキャン。                                    |
| 初期版の対象外 | 常時録音、完全自律送信、購入、組織管理、外部サービス全履歴の自動走査。                                       |

## 7. エンドツーエンド・ユーザーフロー

### 7.1 共通Action Lifecycle

| **段階**      | **状態**                                    | **ユーザーに見えるもの**               |
|---------------|---------------------------------------------|----------------------------------------|
| 1\. Ingest    | Sourceを取得・正規化。                      | 対象期間、ソース数、取得状態。         |
| 2\. Detect    | Open Loop候補を抽出。                       | 候補件数（まだレビュー対象ではない）。 |
| 3\. Reconcile | 重複、部分完了、Closed、更新、矛盾を処理。   | 統合数、部分完了数、Closed数、追加確認数。 |
| 4\. Propose   | Action Proposalを作成。                     | 根拠、実行内容、リスク、Confidence。   |
| 5\. Review    | Approve / Dismiss / Edit / Snooze。         | スワイプカードと詳細。                 |
| 6\. Execute   | 承認内容に従って実行。                      | 進行状態、外部影響、エラー。           |
| 7\. Receipt   | 実行結果を元のCommitmentへ結び付ける。      | 満たした条件、残条件、状態、Undo、元Evidence。 |
| 8\. Re-scan   | 将来スキャンでReceiptを状態Evidenceとして利用。 | Closedは再表示せず、Partially closedは残条件だけを提示する。 |

#### 実行キューとUndo

- 実行を選択したActionは、直ちには実行せず、30秒間の猶予を持つ実行キューへ入れる。
- 実行方法は、内容に依存しない2種類とする。`そのまま実行`は現在のPreviewでキューへ入れ、`指示を追加して実行`は追加指示をPreviewへ反映してからキューへ入れる。
- 猶予中はUndoでキューから取り消し、実行するかどうかを判断する状態へ戻せる。実行後の複雑なロールバックは初期版の対象外とする。
- `実行しない`としたActionは、いつでもUndoで実行可否を判断する状態へ戻せる。

### 7.2 Public Demo Mode

1. 公開URLを開く。ログイン、APIキー入力、ファイル準備は不要。
2. 「Try a sample workday」を選択する。
3. Codexスレッド3件、明示インポートされた文字起こし1件、外部会話ログ1件を読み込む。
4. 「5 sources / 7 candidates / 2 duplicates merged / 1 execution completed and partially closed / 3 ready」を表示する。
5. カードを右スワイプして承認、左で却下、上またはEditで不足情報を補う。
6. サイト内のDemo Gmail Draft、Demo Meeting Draft、Demo Researchへモック実行する。
7. Action ReceiptとUndoを確認する。

### 7.3 Codex Plugin Mode

1. ユーザーがOpen Loop InboxのSkill／Pluginを明示起動する。
2. 対象期間とソースを選ぶ。既定は直近7日のCodexスレッドと、ユーザーが選んだインポート済みログ。
3. ローカル連携がCodex App Serverからスレッド一覧とターン履歴を取得する。
4. 共通Coreが候補検出・照合・Action Proposal生成を行う。
5. Codex内ではAction一覧を構造化して提示し、番号または承認コマンドで判断する。
6. 承認されたResearchを、Plugin専用ローカル保存先にMarkdownレポートとして実行する。Gmail Draftと会議ドラフトは外部アカウントへ書き込まず、Demo MockとしてReceiptを生成する。
7. Receiptを保存し、次回の照合に利用する。

### 7.4 代表的なプロダクトケース

| **ケース**       | **入力**                                                             | **期待動作**                                                               |
|------------------|----------------------------------------------------------------------|----------------------------------------------------------------------------|
| Duplicate Merge  | 外部会話ログで「Cursor受賞例を調べたい」、文字起こしで「5件、月曜までに共有」。 | 一つのCommitmentへ統合し、より具体的な件数と期限を採用する。両方のEvidenceを表示する。 |
| Partially Closed | 文字起こしで「エラーハンドリングを追加してレビュー依頼」、後続Codexで実装とテストが完了。 | 実装は完了として扱い、残りのレビュー依頼だけをActionとして表示する。 |
| Needs Input      | 「来週田中さんと30分話す」。                                         | 日時不足を検知し、一問だけ候補時間を確認。勝手に招待しない。               |
| Executable Draft | 「佐藤さんへ問題ないと返す」。                                       | Gmail下書きまたはDemo Gmailへ下書きを作成。送信しない。                    |

## 9. UX・画面要件

### 9.1 UX原則

| **原則**                   | **要件**                                                             |
|----------------------------|----------------------------------------------------------------------|
| Reduce, don’t dump         | AIは候補を増やすのではなく、重複・完了を処理して少数へ減らす。       |
| Evidence before confidence | Confidence数値より先に、元発言とSourceを確認できる。                 |
| Preview before effect      | 承認前に「どのアプリで何が起きるか」を明示する。                     |
| One question, not a chat   | 不足情報はカード内の一問・選択肢で解決し、長いチャットを開始しない。 |
| Fast path + safe path      | 低リスクは一動作、中リスクはDraft、高リスクは対象外または二段階。    |
| Reversible by default      | 可能な操作はUndoを提供し、Receiptに戻し方を残す。                    |

### 9.2 情報アーキテクチャ

| **画面**           | **主目的**                 | **必須要素**                                         |

| Action Deck        | 高速判断。                 | カード、進捗、操作、Undo。                           |

| Execution Progress | 実行状態を明示。           | 実行先、現在状態、Cancel／Retry。                    |
| Receipt Summary    | 結果確認。                 | 実行済み、下書き、却下、保留、Undo。                 |

| **領域**       | **表示内容**                                     |
|----------------|--------------------------------------------------|
| ヘッダー       | Action type、Risk、Status。                      |
| 主文           | ユーザーが判断する具体的Action。動詞から始める。 |

### 9.4 スワイプ挙動

| **操作**         | **意味**                   | **視覚フィードバック**                             |
|------------------|----------------------------|----------------------------------------------------|

| 左               | 却下。                     | 却下理由の任意選択。Undo可能。                     |

### 9.5 UIコピー要件

- 「AIが判断した」ではなく、「なぜそう判断したか」を短文で示す。
- 「実行します」ではなく、「Gmail下書きを作成。送信しません」のように具体化する。
- 「低・中・高リスク」だけでなく、リスクの理由を示す。
- Closed除外とPartially closedへの更新は隠さず、CodexのどのEvidenceを使ったかサマリーで示す。
- エラー時は技術用語より、ユーザーが次に取れる行動を示す。

### 9.6 アクセシビリティ

- スワイプのみを必須にしない。すべての操作にボタンとキーボード代替を用意。
- Risk、Status、Sourceを色だけで区別しない。テキストまたはアイコンを併用。
- フォーカス順、スクリーンリーダーラベル、十分なコントラストを確保。
- カード移動アニメーションは短くし、Reduced Motion設定に対応。

## 10. データ・ドメイン要件

### 10.1 主要エンティティ

| **エンティティ**      | **目的**                   | **主要属性**                                        |
|-----------------------|----------------------------|-----------------------------------------------------|
| Source                | 取得元の会話・履歴を識別。 | 種別、ID、タイトル、作成／更新時刻、アクセス範囲。  |
| Conversation Artifact | Source固有形式を共通化。   | Turns、実行結果、メタデータ、原文。                 |
| 根拠箇所         | 判断の根拠。               | 情報源、発言位置、原文引用、時刻、根拠の種別。            |
| 対応候補      | 検出直後の未照合候補。     | 行為、担当者、期限、対象、根拠、根拠の十分さ。     |
| 照合・整理グループ  | 同一または関連候補の集合。 | 候補の構成員、関係、最新状態、矛盾。                  |
| 対応案       | ユーザーが判断する単位。   | タイトル、状態、実行上の注意度、実行前の確認内容、根拠、理由。    |
| Approval Decision     | 人間の判断。               | Approve、Dismiss、Edit、Snooze、時刻、変更差分。    |
| 作業実行の対応記録     | 実行結果と対応事項の状態の監査。 | 実行機能、実行内容の要約、結果、満たした条件、残条件、外部ID、取り消し、エラー。 |

### 10.2 Statusモデル

作業実行の進行状態と対応事項の完了状態を分けて保持する。
一つの実行が成功しても、元の対応事項に残条件があれば完了済みにはしない。

#### Execution Status

| **Status**          | **意味**                         |
|---------------------|----------------------------------|
| Not started         | まだ実行を開始していない。       |
| Executing           | Codexまたは接続アプリが実行中。  |
| Execution completed | 指定した実行が検証まで完了した。 |
| Failed              | 実行または検証に失敗した。       |

#### 対応事項の状態

| **Status**          | **意味**                                                   | **レビュー対象** |
|---------------------|------------------------------------------------------------|------------------|
| 未完了                | 対応事項を検出したが、まだ判断していない。              | はい             |
| 承認待ち  | 実行案が揃い、人間の承認を待つ。                           | はい             |
| 追加情報が必要         | 日時、相手、範囲などの重要情報が不足する。                 | はい             |
| 条件の矛盾            | 複数の根拠で条件が矛盾する。                             | はい             |
| 一部完了    | 一部条件は満たされたが、残作業がある。                     | はい             |
| 完了済み              | 元の対応事項を満たす根拠と必要な承認が揃った。       | いいえ           |
| 対応不要           | ユーザーが実行不要と判断した。                             | いいえ           |
| 要再対応            | 失敗、差し戻し、条件変更によって再対応が必要になった。     | はい             |
| 重複統合済み           | 別の対応事項へ統合済み。                                 | いいえ           |

### 10.3 Provenance要件

- Actionのすべての事実値は、Evidenceまたはユーザー編集へ追跡できること。
- AI推定値はEvidence由来の事実値と区別すること。
- 複数Source統合後も、各Sourceの原文・時刻・関係を失わないこと。
- 対応事項の状態の判定には、ユーザー宣言、Codex実行、テスト結果、外部反映、対応記録のどれが根拠になったかを保持すること。

### 10.4 時系列・優先規則

| **規則**                               | **内容**                                             |
|----------------------------------------|------------------------------------------------------|
| Latest explicit update wins            | 後続の明示的な期限・数量・対象更新を優先する。       |
| Execution evidence outranks intent     | 実行結果・テスト成功は、過去の「やる予定」より強い。 |
| User decision outranks model inference | ユーザーの編集・却下・承認をAI推定より優先する。     |
| Conflict remains visible               | 同等に強い矛盾証拠がある場合は勝手に解決しない。     |
| Receipts update commitments              | Receiptは満たした条件と残条件を更新し、全条件が揃った場合だけCommitmentをClosedにする。 |

## 11. AI判断・照合ロジック要件

### 11.1 判断パイプライン

| **処理**               | **入力**                 | **出力**                   | **失敗時**               |
|------------------------|--------------------------|----------------------------|--------------------------|
| Detection              | 正規化Conversation       | Candidate Action           | 低Confidenceとして隔離。 |
| Entity resolution      | 人物、日時、対象         | 正規化属性                 | Unknown／Needs input。   |
| Clustering             | 候補群                   | 関連グループ               | 統合せず別カード。       |
| State resolution       | Evidence時系列           | Ready／Partially closed／Closed／Conflict | Conflictとして人間確認。 |
| Proposal generation    | 解決済みAction           | カード表示・実行Preview    | 実行不可として表示。     |
| Execution verification | 承認Payloadと実行Payload | Proceed／Re-approve        | 実行停止。               |

### 11.2 検出ポリシー

- メインキューは高精度を優先し、曖昧な願望・雑談を積極的に除外する。
- Action ownerがユーザー本人または明示担当者でない場合、本人ToDoとして出さない。
- 外部影響のあるActionは、相手・対象・時間・実行形式が不足していればNeeds input。
- 「調べたい」「比較したい」はResearch候補になり得るが、単なる興味と実行意図を区別する。
- Codexスレッドでは、ユーザー依頼だけでなくCodexの完了報告、テスト結果、ファイル変更を状態証拠として使用する。

### 11.3 重複判定基準

| **観点** | **同一判定に使う情報**                                              |
|----------|---------------------------------------------------------------------|
| 目的     | 最終成果物または解決したい問題が同じ。                              |
| 対象     | 同じ相手、プロジェクト、ファイル、テーマ。                          |
| 時期     | 同一期間または後続更新として自然。                                  |
| 行為     | 「調べる」「5件比較して共有」など抽象度が違っても同じ成果へ向かう。 |
| 除外条件 | 同じテーマでも成果物・担当・期限が独立している場合は統合しない。    |

### 11.4 完了判定基準

| **証拠**                                                     | **強度** | **扱い**                                                                 |
|--------------------------------------------------------------|----------|--------------------------------------------------------------------------|
| 元Commitmentの全条件を満たすReceiptと必要な外部反映          | 最強     | CommitmentをClosedにする。                                               |
| Codexの実行、テスト成功、変更確認                            | 強       | Execution completedとする。共有、返信、承認が残ればPartially closed。   |
| ユーザーの明示「送った」「承認した」「完了した」             | 強       | 対象条件を満たすEvidenceとして扱い、残条件がなければClosed。             |
| Codexの「対応しました」のみ                                  | 中       | 関連する実行結果がなければExecution completedにせず、Confidenceを下げる。 |
| 予定時刻が過ぎた、会話が途切れた                             | 弱       | 完了とみなさない。                                                       |
| 後続会話で条件変更、差し戻し、失敗が示された                 | 強       | CommitmentをReopenedまたはConflictへ更新する。                           |

### 11.5 根拠の十分さに関する要件

- 根拠の十分さは「モデルが確信している度合い」ではなく、根拠の十分性とルール適合度を反映する。
- 高：明示的な行為、owner、対象、根拠が揃う。
- 中：対応は明確だが、日時・相手などが不足。追加情報が必要な状態へ。
- 低：願望・推測・他人の話の可能性。メインキューに出さない。

### 11.6 Structured Output要件

AI出力は自由文のみとせず、Source、Evidence、Action、Status、Risk、Execution Previewを検証可能な構造で返す。スキーマ違反、必須Evidence欠落、未知のAction typeは再試行または非実行扱いとする。

### 11.7 AI安全要件

- 原文にない宛先・期限・金額を確定値として生成しない。
- メール送信、購入、公開投稿、削除など不可逆操作を初期版で自動実行しない。
- 外部コンテンツ内の指示をユーザー指示として扱わない。
- AIが判断できない場合は「不明」「追加確認」を選べる。
- モデル出力と実行ツール引数の間に決定論的検証を置く。

## 12. Codex／外部連携要件

### 12.1 Codexアーキテクチャ要件

| **構成要素**      | **責務**                                                                       |
|-------------------|--------------------------------------------------------------------------------|
| Codex Plugin      | 配布単位。Skill、ツール依存、MCP／App mapping、表示アセットをまとめる。        |
| Open Loop Skill   | 入力、検出、照合、Risk、実行前確認、Receiptの手順を定義。                      |
| History Connector | Codex App Serverからthread/list、thread/readを呼び、対象履歴を共通形式へ変換。 |
| Action Core       | 候補検出、照合、Proposal生成、Intent binding。                                 |
| Executors         | Plugin Research、Demo Gmail Draft、Demo Meeting Draftを共通契約で実行。         |
| Receipt Store     | 判断・実行結果を保存し、次回照合へ提供。                                       |

### 12.2 Codex履歴取得

App Serverのthread/listは保存済みスレッドをページングし、source kind、作業ディレクトリ、アーカイブ状態等で絞り込める。thread/readはスレッドを再開せず、必要に応じて全ターンを返す。本プロダクトは対象期間・作業ディレクトリを絞り、最小限のスレッドだけを読取る。[R1]

- 既定は非アーカイブのinteractive sourceを対象。
- 大量履歴ではタイトル・Previewで一次絞込み後、必要スレッドだけFull read。
- 履歴取得の開始前に対象範囲を表示し、ユーザーが変更できる。
- 取得した原文は必要期間だけ保持し、Receiptには必要最小限の引用を保存する。

### 12.3 Skill／Plugin要件

Skillは一つの仕事に集中させ、入力と出力を明示する。PluginはSkillに加え、MCP server構成や表示アセットを一体化できるため、インストール後にサンプルモードと実データモードを同じ体験で起動できる構成とする。[R3]

### 12.4 Executor別要件

| **Executor**  | **初期版動作**                                          | **安全境界**                              |
|---------------|------------------------------------------------------|-------------------------------------------|
| Plugin Research | 指定テーマを調査し、短い出典付きMarkdownレポートをPlugin専用ローカル保存先へ作成。 | 外部公開なし。保存先をPreviewとReceiptに明示。 |
| Demo Gmail Draft | 宛先・件名・本文の下書き結果を画面内に作成。 | Gmailアカウントへ接続せず、送信しない。 |
| Demo Meeting Draft | 自分用の仮予定結果を画面内に作成。 | Calendarアカウントへ接続せず、参加者を招待・通知しない。 |

### 12.5 Intent-bound Approval

> **必須安全設計**
>
> ユーザーが承認したカードの意味と、実際のツール実行内容を結びつける。宛先、日時、Draft/Send、対象Repository、削除有無など重要属性が変わった場合は、承認を無効化して再確認する。

| **比較対象**         | **差分時の挙動**                                        |
|----------------------|---------------------------------------------------------|
| 宛先・参加者         | 必ず再承認。                                            |
| 日時・期限           | ユーザー影響があれば再承認。                            |
| DraftからSend        | 初期版では拒否。                                        |
| 参照範囲・Repository | 範囲拡大時は再承認。                                    |
| 本文表現の軽微な整形 | 意味が変わらない範囲は許可し、Receiptに最終内容を残す。 |

### 12.6 Sites要件

ChatGPT SitesはWebサイト・Webアプリ・ゲームを作成、ホスト、共有でき、既存の互換プロジェクトから本番URLを発行できる。公開ベータであり、利用可否はプラン、地域、Workspace設定に依存する。SecretsはSite設定で管理できる。[R4]

- 第一配備先はChatGPT Sitesとする。
- 公開範囲：ログインなしでアクセス可能な共有URL。公開前にシークレットブラウザと別端末で確認する。
- Secrets：OpenAI API key等はSites設定へ保存し、リポジトリ・プロンプト・画面に含めない。
- Fallback：Sites公開が使えない場合に備え、同一アプリを別ホストへデプロイできる構造を保つ。

## 13. 非機能要件

### 13.1 性能

| **項目**           | **初期版目標**                                              |
|--------------------|----------------------------------------------------------|
| 公開デモサイト初期表示 | 3秒以内（一般的なブロードバンド環境）。                  |
| サンプル解析       | Live AIで15秒以内、Fallbackで1秒以内。                   |
| スワイプ反応       | 100ミリ秒以内に視覚反応。                                |
| モック実行         | 2秒以内にReceipt。                                       |
| Codexスキャン      | 直近20スレッドの一次一覧を10秒以内、詳細解析は進捗表示。 |

### 13.2 信頼性

- 公開デモサイトはLive AI失敗時にFallbackへ自動切替し、主要フローを完了できる。
- ソースごとの失敗を隔離し、成功したSourceの結果を失わない。
- 実行中断後もAction状態をUnknownにせず、ExecutionはFailed、CommitmentはOpenまたはReopenedへ明示遷移する。
- 二重クリック・二重スワイプで同じ外部Actionを重複作成しない。

### 13.3 可搬性・配備

- 単一リポジトリでCodex Plugin、Shared Core、公開デモサイトを管理する。
- 公開デモサイトはSitesを第一候補にするが、別ホストにも同じビルド成果を配備できる。
- ローカル開発と公開環境で同じAction Schema・サンプルデータを使用する。

### 13.4 可観測性

| **イベント**   | **記録内容**                             |
|----------------|------------------------------------------|
| Source ingest  | 成功／失敗、件数、期間。                 |
| Detection      | 候補数、低Confidence除外数。             |
| Reconciliation | Merge、Partially closed、Closed、Conflictの件数と理由。 |
| Review         | Approve、Dismiss、Edit、Undo。           |
| Execution      | 開始、承認、成功、失敗、外部ID。         |
| Fallback       | Live AIからFallbackへ切替えた事実。      |

### 13.5 コスト

- 公開デモサイトはサンプルデータを短く保ち、1セッションのモデル呼出し回数を限定する。
- Codex履歴は全件全文ではなく、一覧→候補絞込み→詳細読取の段階処理を行う。
- 同じSource内容にはキャッシュまたはFingerprintを利用し、再解析を避ける。

### 13.6 ローカライゼーション

- 初期版UIは日本語を主言語とし、Action type等の内部用語は英語併記可。
- サンプルデータは日本語を中心にし、将来の多言語対応に備えて英語版を追加できる構造にする。
- 日時解釈はAsia/Tokyoを既定とし、タイムゾーンを表示する。

## 14. セキュリティ・プライバシー・信頼要件

### 14.1 基本原則

| **原則**              | **要件**                                                         |
|-----------------------|------------------------------------------------------------------|
| Explicit scope        | どのSource、期間、Project、Codexディレクトリを読むかを事前表示。 |
| Data minimization     | 必要な会話だけを取得し、不要な全文を長期保存しない。             |
| Evidence transparency | AI判断の原文根拠をユーザーが確認可能。                           |
| Human approval        | 外部影響のある実行は明示承認後。                                 |
| Least privilege       | GmailはDraft、CalendarはHold等、最小権限の操作を選ぶ。           |
| Auditability          | 判断と実行のReceiptを残す。                                      |

### 14.2 データ分類

| **データ**               | **機密度** | **扱い**                                        |
|--------------------------|------------|-------------------------------------------------|
| Codex履歴と明示インポート原文 | 高         | 対象を明示選択し、必要期間だけ扱い、公開デモへ送らない。 |
| Evidence引用             | 高〜中     | 最小限の範囲を保存・表示。                      |
| Action Proposal          | 中         | 原文と分離して保存。                            |
| Receipt                  | 中         | 外部ID・実行結果を保持し、秘密値を含めない。    |
| サンプルデータ           | 低         | 架空人物・架空案件のみ。                        |
| API key／OAuth token     | 最高       | Secrets管理。ログ、リポジトリ、画面へ出さない。 |

### 14.3 外部書込み安全性

- メール送信、購入、公開投稿、削除、契約、支払は初期版で実行しない。
- GmailやCalendarの実アカウントは初期版で接続しない。Gmail Draftと会議ドラフトはDemo Mockとして扱う。
- 中リスク操作はDraft／Tentativeとして作成し、外部通知を抑止する。
- ユーザー承認と実行Payloadの重要属性を比較する。
- 実行結果をReceiptに表示し、可能ならUndoを提供する。
- Prompt injectionを含む外部コンテンツはEvidenceとして扱い、権限変更指示として実行しない。

### 14.4 Codex履歴と外部会話ログの扱い

- Codex履歴は対象期間と作業ディレクトリを選び、全履歴の無差別走査を既定にしない。
- 会議文字起こしやChatGPTエクスポートなどの外部会話ログは、ユーザーが対象を確認して明示インポートする。
- 外部サービスの認証済み全履歴へ自動アクセスする機能を初期版の前提にしない。
- Sourceを削除した場合、そのSource由来Evidenceと未実行Proposalを削除できる。
- 公開デモサイトは架空のサンプルだけを使い、実ユーザー履歴のアップロードを既定フローにしない。

## 15. 評価・受入基準

### 15.1 ゴールデンデータセット

初期版の評価には、少なくとも20件の短い会話断片と、5件の複数ソースシナリオを用いる。各シナリオには正解Action、owner、期限、重複関係、完了状態、Evidenceを人手で付与する。

| **ケース種別**  | **最低件数** | **目的**                 |
|-----------------|--------------|--------------------------|
| 明示Action      | 6            | 基本検出。               |
| 非Action／雑談  | 5            | False positive抑制。     |
| Duplicate       | 3組          | クロスソース統合。       |
| Partially closed / Closed later | 3組 | 後続Evidenceによる残条件更新と完了判定。 |
| Needs input     | 3            | 不足情報の検出。         |
| Conflict        | 2            | 勝手に解決しないこと。   |

### 15.2 品質指標

| **指標**             | **計算**                                   | **受入目標** |
|----------------------|--------------------------------------------|--------------|
| Detection Precision  | 妥当な候補／表示候補。                     | 85%以上      |
| Detection Recall     | 検出した正解／全正解。                     | 75%以上      |
| Merge Precision      | 正しく統合した組／全統合。                 | 90%以上      |
| Closure Precision    | 正しくClosedまたはPartially closedにした件／全closure判定。 | 90%以上 |
| Evidence Validity    | 原文が判断を支持するAction／全Action。     | 100%         |
| Payload Fidelity     | 承認内容と実行内容が一致。                 | 100%         |

### 15.3 初期版の受入テスト

| **テスト**                  | **合格条件**                                                |
|-----------------------------|-------------------------------------------------------------|
| AT-01 One-click trial        | 公開URLから2分以内にReceiptまで到達する。                       |
| AT-02 Cross-source merge    | 明示インポートされた文字起こしと外部会話ログの同一Researchを一件へ統合。 |
| AT-03 Partial closure        | Codexの実装とテスト完了を使い、残っている共有Actionだけを表示。        |
| AT-04 Needs input           | 日時不足を検出し、候補時間の一問を表示。                    |

| AT-11 Queue grace period    | 実行選択後30秒以内のUndoで実行されず、Actionが実行可否の判断状態へ戻る。 |

### 15.4 ユーザーテスト観察項目

- 初見でも「Try sample」から開始できるか。
- カードを見て、何が起きるか10秒以内に説明できるか。
- 統合、部分完了、Closed除外が単純ToDo抽出との差として伝わるか。
- 右スワイプへの不安がEvidence・Preview・Riskで解消されるか。
- Codexの役割が、開発支援ではなく履歴・実行・Receiptをつなぐ閉ループとして理解されるか。

## 16. リスクと対策

### 16.1 主要リスク

| **リスク** | **影響** | **対策** |
|---|---|---|
| 単純なToDo抽出に見える | 差別化が伝わらない。 | 体験の中心を、候補抽出ではなく、重複統合、部分完了、Closed除外によるレビュー件数の削減に置く。 |
| Codexの必然性が伝わらない | 汎用LLMと外部APIでも同じに見える。 | Codex履歴を一次Sourceとして読み、Execution Evidenceの取得、残作業の実行、Receiptによる状態更新までを一つのloopとして示す。 |
| 外部履歴アクセスを過大に受け取られる | プライバシーと実現性への信頼を損なう。 | Codex履歴は選択範囲だけを直接参照し、外部会話ログはユーザーが内容を確認して明示インポートする。 |
| False positiveが多い | Inbox自体が新しい負担になる。 | 再現率より適合率を優先し、低Confidence候補をメインキューから外す。 |
| 誤ったClosed判定 | 重要なActionが見えなくなる。 | Execution StatusとCommitment Statusを分離し、元の約束に残条件があればPartially closedとして保持する。 |
| 不適切な重複統合 | 独立したActionが失われる。 | 成果物、owner、対象、期限を比較し、曖昧な場合は統合せず、Conflictまたは人間確認へ送る。 |
| 誤実行 | 外部への誤送信や不正な変更につながる。 | Draft／Hold、Intent-bound Approval、冪等性、実行Preview、Receipt、Undoを組み合わせる。 |
| スワイプが表面的なギミックに見える | UXの必要性が弱くなる。 | Evidence、残条件、Risk、実行Previewを一枚に集約し、Agent提案の高速レビュー手段として位置付ける。 |
| 履歴量の増加で処理が遅くなる | 日常利用に耐えない。 | 一覧取得、候補絞込み、詳細読取の段階処理とFingerprintによる再解析回避を行う。 |

### 16.2 最大の製品リスク

ユーザーは、AIが候補を見落とすことよりも、AIが重要なActionを誤って完了扱いしたり、別のActionへ統合したりすることを強く恐れる可能性がある。

初期版では削減率だけを追わず、除外理由の透明性、元Evidenceへの遡及、Undo、強い完了証拠を優先する。

| 未完了事項             | Codexスレッドまたは明示的に取り込んだ会話で発生し、完了済み、対応不要、委任済みのいずれも確認できない対応事項。 |

| 根拠箇所         | 対応の判断を支持する原文引用と情報源内の位置。                               |
| Execution Preview     | 承認すると、どのツールで何が作成・変更されるかの事前説明。                 |

| 対応記録        | 元の対応事項、ユーザー判断、実行内容、満たした条件、残条件、時刻、外部ID、取り消しを結び付けた実行・判断記録。 |
| 個別作業の実行        | エージェントへ渡した個別作業が、実行と検証を終えるまでの流れ。                    |
| 対応完了までの流れ       | 元の対応事項が、関連作業、承認、共有、返信、外部反映まで満たされるまでの流れ。    |
