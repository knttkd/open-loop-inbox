# Open Loop Inbox（未完了事項インボックス）実装フェーズ

このディレクトリは、[プロダクト要件](../docs/プロダクト要件.md)を実装へ落とし込むための実行計画である。

## 進め方の原則

- 公開デモとCodexプラグインで、同じスキーマ、サンプルデータ、照合・整理のルールを使う。
- 最初にフォールバックデータで一連の動作を完成させ、その後ライブAIを接続する。
- 作業実行の状態と対応事項の状態を常に分ける。
- すべての対応と状態判定を根拠へ追跡可能にする。
- 外部影響のある操作は、実行前の確認内容、承認、実行内容の照合、対応記録を通す。
- 各フェーズの完了条件を満たすまで、依存する次フェーズを完了扱いにしない。

## 全体ロードマップ

| フェーズ | 名称 | 主な成果 | 完了の判断 |
|---|---|---|---|
| 0 | 方針・スコープ固定 | 技術方針、構成、要件対応表 | 実装境界とMVPの定義が合意済み |
| 1 | 共通基盤 | スキーマ、状態遷移、サンプル、照合テスト | 代表5ケースが決定論的に通る |

| 3 | デモ実行機能・対応記録 | モック実行、安全検証、対応記録、取り消し | 外部認証なしで対応完了までの流れを体験できる |
| 4 | ライブAI | 検出、抽出、照合・整理、フォールバック | ゴールデンデータの品質目標を満たす |

## 推奨マイルストーン

### M1：Core Contract完成

フェーズ0〜1。データモデルと代表シナリオを固定し、UIやAIに依存せず照合・整理の結果を再現できる状態。

### M2：公開デモの縦切り完成

フェーズ2〜3。回収 → 照合・整理 → 確認・判断 → 実行 → 対応記録をフォールバックデータだけで完走できる状態。ハッカソンの最小デモ成立点とする。

### M3：動的解析・Codex接続

フェーズ4〜5。ライブAIとCodex実履歴を共通基盤へ接続し、実データから対応を作れる状態。

### M4：実行・公開品質

フェーズ6〜7。少なくとも1種類の実行機能を動かし、公開URLと受入試験を完了する。

## 依存関係

```text
フェーズ0 → フェーズ1 → フェーズ2 → フェーズ3
                     └→ フェーズ4 → フェーズ5 → フェーズ6
フェーズ2〜6 → フェーズ7
```

フェーズ2とフェーズ4は、フェーズ1完了後に並行作業できる。ただしフェーズ4の出力スキーマはフェーズ1の契約に従う。

## 全体の完了条件

- 公開URLからログインなし、外部ファイルなしで2分以内に対応記録へ到達できる。
- Codex Sitesの変更可能なアドレス部分は `open-loop-inbox` とし、個人を識別できる文字列を含めない。
- Codex 3件、文字起こし1件、外部会話1件を共通モデルで処理できる。
- 重複統合済み、一部完了、完了済みによる除外、追加情報が必要な状態を根拠付きで示せる。
- スワイプ、ボタン、キーボードで同じ判断ができ、取り消しできる。
- 承認した実行内容と実際の実行内容の重要属性が一致し、差分時は再承認される。
- ライブAI障害時もフォールバックで主要フローを完了できる。
- 実際のCodex履歴から対応を作り、調査またはCodexタスクを実行できる。
- 要件書のAT-01〜AT-10がすべて合格する。
