# Phase 7：品質保証・公開・計測

## 目的

プロダクトを公開可能な品質へ仕上げ、価値仮説、安全性、性能を継続的に確認できる状態にする。

## 対象要件

非機能要件、セキュリティ・プライバシー要件、評価・受入基準、FR-SITE全般、AT-01〜10。

## やること

### 受入試験

- [ ] AT-01 One-click trial
- [ ] AT-02 Cross-source merge
- [ ] AT-03 Partial closure
- [ ] AT-04 追加情報が必要
- [ ] AT-05 Swipe parity
- [ ] AT-06 Mock execution
- [ ] AT-07 Codex live scan
- [ ] AT-08 Real execution
- [ ] AT-09 Safety
- [ ] AT-10 Fallback

### 品質・安全性

- [ ] モバイル、デスクトップ、主要ブラウザで確認する。
- [ ] キーボードとスクリーンリーダーを確認する。
- [ ] Reduced Motionとコントラストを確認する。
- [ ] モデル障害、Source失敗、Executor失敗を注入する。
- [ ] 二重操作と冪等性を確認する。
- [ ] Prompt injectionを含む入力を確認する。
- [ ] 秘密値が画面、ログ、リポジトリ、ビルド成果物にないことを確認する。
- [ ] シークレットブラウザと別端末で公開フローを確認する。

### 性能

- [ ] 初期表示3秒以内を測定する。
- [ ] Fallback解析1秒以内を測定する。
- [ ] Live AI解析15秒以内を目標に計測する。
- [ ] スワイプの視覚反応100ms以内を確認する。
- [ ] モック実行2秒以内を確認する。
- [ ] Codex一覧20件を10秒以内で取得できるか確認する。

### 可観測性

- [ ] Source ingestの成功、失敗、件数、期間を記録する。
- [ ] 検出候補数と根拠の十分さが低いため除外した数を記録する。
- [ ] 統合、一部完了、完了済み、条件の矛盾の件数と理由を記録する。
- [ ] Approve、Dismiss、Edit、Undoを記録する。
- [ ] Executionの開始、成功、失敗、外部IDを記録する。
- [ ] Fallback切替を記録する。
- [ ] 原文や秘密値を分析イベントへ含めない。

### 公開と運用

- [ ] Sitesまたは代替ホストへ公開する。
- [ ] Codex Sitesで変更可能なアドレス部分は `open-loop-inbox` とする。
- [ ] 公開URLに個人名、個人のユーザー名、メールアドレス由来の文字列（例：`kanta.takeda`）を含めない。
- [ ] Secretsをホスト設定へ登録する。
- [ ] READMEに起動、テスト、Plugin導入、デモ手順を書く。
- [ ] Resetとサンプルの再実行を確認する。
- [ ] 障害時のFallbackとロールバック手順を用意する。

## ユーザーテスト

- 初見でTry sampleから開始できるか。
- カードを見て10秒以内に何が起きるか説明できるか。
- 重複統合済み、一部完了、完了済みによる除外が単純なToDo抽出との差として伝わるか。
- 根拠、実行前の確認内容、実行上の注意度が承認への不安を下げるか。
- Codexの履歴→判断→実行→Receiptの閉ループが理解されるか。
