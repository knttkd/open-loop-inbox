# Phase 0：方針・スコープ固定

## 目的

プロダクト要件を実装可能な境界へ変換し、MVPで作るもの、後回しにするもの、各コンポーネントの責務を固定する。

## やること

- [ ] Web、Shared Core、Pluginを管理できるモノレポ構成を決める。
- [ ] Webフレームワーク、言語、Schema検証、テスト、状態管理、配備先を決める。
- [ ] 公開デモ、共通基盤、Codexプラグイン、実行機能、対応記録ストアの責務を定義する。
- [ ] Must要件をPhaseと受入テストへ対応付ける。
- [ ] 代表ケースの入力と期待結果を確定する。
- [ ] Public Demoの画面遷移と主要コピーを確定する。
- [ ] Live AIと事前生成Fallbackの切替方式を決める。
- [ ] 外部書込み、秘密情報、履歴取得範囲の安全境界を記述する。
- [ ] 対象外機能をバックログへ分離する。

## 推奨リポジトリ構成

```text
apps/web                 公開デモ
packages/schemas         共通Schema
packages/core            検出後の照合・状態解決
packages/sample-data     サンプルとFallback
packages/executors       Executor共通契約
plugin/open-loop-inbox   Codex PluginとSkill
tests/golden             AI・照合評価データ
```

## 決めるべき契約

- Source固有形式をShared Coreへ持ち込まない。
- AI出力はSchema検証後にだけ下流へ渡す。
- ExecutorはPreview、実行、検証、Undo情報を共通形式で返す。
- 対応記録は成功時だけでなく、対応不要、重複統合済み、完了済みにも作る。
- 原文、抽出事実、AI推定、ユーザー編集を別フィールドで保持する。

## 成果物

- 技術選定記録
- アーキテクチャ概要
- 要件トレーサビリティ表
- 画面遷移
- 代表ケース仕様
- 初期バックログ

## 完了条件

- [ ] 初期版の範囲と対象外が曖昧でない。
- [ ] 各Must要件に実装Phaseと検証方法がある。
- [ ] 代表ケースで期待する対応、状態、根拠、対応記録が定義済み。
- [ ] Phase 1で実装するSchema境界が確定している。

## 次Phaseへの移行条件

データモデルの作成中にUIや特定Executorの都合で頻繁に構造変更しなくてよい程度まで、責務境界が固まっていること。
