# Phase 1：Shared Coreとゴールデンデータ

## 目的

Public DemoとCodex Pluginが共有するデータモデル、状態遷移、照合規則、評価データを作る。

## 対象要件

FR-NRM-001〜005、FR-REC-001〜009、FR-ACT-001〜004、FR-STA-001〜002、データ・ドメイン要件、Provenance要件。

## やること

### Schema

- [ ] Sourceを定義する。
- [ ] Conversation ArtifactとTurnを定義する。
- [ ] 根拠箇所と根拠の種別を定義する。
- [ ] 対応候補を定義する。
- [ ] 照合・整理グループを定義する。
- [ ] 対応案を定義する。
- [ ] Approval Decisionを定義する。
- [ ] Execution Receiptを定義する。
- [ ] 作業実行の状態と対応事項の状態を別々に定義する。
- [ ] 全Schemaにバージョンを持たせる。

### 状態と照合

- [ ] Latest explicit update winsを実装する。
- [ ] 作業実行の根拠を予定より優先するルールを実装する。
- [ ] User decision outranks model inferenceを実装する。
- [ ] 同等に強い矛盾をConflictとして残す。
- [ ] Duplicate統合後も全Evidenceを保持する。
- [ ] 完了条件を条件単位で評価し、残条件を生成する。
- [ ] Closedをレビューキューから除外する。
- [ ] 各判定に機械可読な理由コードと表示用理由を持たせる。

### サンプルと評価データ

- [ ] Codexスレッド3件以上を作る。
- [ ] 文字起こし1件以上を作る。
- [ ] 外部会話ログ1件以上を作る。
- [ ] Duplicateを2件以上含める。
- [ ] 一部完了と完了済みを含める。
- [ ] 追加情報が必要と条件の矛盾を含める。
- [ ] 20件以上の短いゴールデン会話断片を用意する。
- [ ] 期待Action、owner、期限、関係、状態、Evidenceを人手で付与する。

## 必須代表ケース

| ケース | 期待結果 |
|---|---|
| 重複統合 | 抽象的な依頼と具体化された依頼を一件へ統合し、すべての根拠を残す |
| 一部完了 | 実装・テスト完了後、確認依頼だけを残す |
| 完了済み | 全条件を満たす強い根拠があれば確認対象から除外する |
| 追加情報が必要 | 日時などの重要情報不足を一問の確認へ変換する |
| 条件の矛盾 | 同等に強い矛盾を勝手に解消しない |

## テスト

- Schema validation
- 状態遷移の単体テスト
- 時系列優先規則の単体テスト
- Duplicateの正例・負例
- 一部完了／完了済み／要再対応のテスト
- Evidence欠落時の拒否テスト
- 同じ入力から同じFallback結果が出るスナップショットテスト

## 完了条件

- [ ] 全Actionの事実値がEvidenceまたはユーザー編集へ遡れる。
- [ ] 作業実行の成功だけでは対応事項を完了済みにしない。
- [ ] 代表5ケースが自動テストで通る。
- [ ] Public Demoで利用可能なFallback結果が完成している。
- [ ] Source固有フォーマットを下流が参照しない。
